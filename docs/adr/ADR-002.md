# ADR-002: Combinación Factory + Strategy para la selección del procesador de pedidos

## Estado

Aceptado — 2026-05-16

## Contexto

El sistema debe calcular el costo de un pedido de forma diferente según su tipo:

| Tipo         | Regla de cálculo              |
|--------------|-------------------------------|
| ESTANDAR     | subtotal × 1.1                |
| EXPRESS      | subtotal × 1.3                |
| INTERNACIONAL| subtotal × 1.5 + $25.00       |

La implementación inicial resolvía esto con un bloque `if/else if` en
`ServicioPedidosLegacy.procesarPedido()`. Este diseño tiene tres problemas
concretos:

1. **Viola el Principio Open/Closed (OCP):** agregar el tipo PRIORITARIO requiere
   modificar el servicio existente y añadir una rama al condicional.
2. **Cyclomatic Complexity = 4** en un solo método, con tendencia a crecer.
3. **Baja testabilidad:** no se puede probar el cálculo de un tipo específico
   sin ejecutar toda la lógica del servicio.

Se evaluaron tres alternativas:

- **Alternativa A (status quo):** mantener el `if/else if` — simple pero viola OCP.
- **Alternativa B (switch + enum):** reduce el texto pero sigue en la misma clase,
  el problema de modificación persiste.
- **Alternativa C (Strategy + Factory):** cada tipo tiene su propia clase;
  la selección se delega a un mapa construido por inyección de dependencias.

## Decisión

Se elige la **Alternativa C: Strategy + Factory**.

- La interfaz `ProcesadorPedido` (puerto en `dominio.puertos`) define el contrato:
  `getTipo()` y `procesar(Pedido)`.
- Cada tipo tiene su implementación en `adaptadores.procesadores`:
  `ProcesadorPedidoEstandar`, `ProcesadorPedidoExpress`, `ProcesadorPedidoInternacional`.
- `ProcesadorPedidoFactory` recibe por inyección la lista completa de implementaciones
  y construye un `Map<TipoPedido, ProcesadorPedido>` en el constructor.
- Agregar un nuevo tipo requiere únicamente crear una nueva clase `@Component`
  sin modificar ninguna clase existente.

La regla `procesadoresImplementanPuerto` en `ReglasArquitecturaTest` verifica que
todo componente en el paquete `procesadores` implementa el puerto, previniendo
clases huérfanas sin contrato.

## Consecuencias

**Positivas:**
- La Cyclomatic Complexity de `ServicioPedidos` cae de 4 a 1 (sin condicionales).
- Agregar el tipo PRIORITARIO es agregar una clase, no modificar ninguna existente.
- Cada procesador es testeable de forma aislada con una sola prueba unitaria.
- La regla ArchUnit garantiza que el contrato se mantiene en cada build.

**Negativas:**
- El sistema tiene 3 clases adicionales (una por tipo) donde antes había un método.
  Para equipos pequeños, esto puede percibirse como sobreingeniería inicial.
- La selección implícita por mapa de inyección de dependencias puede sorprender
  a desarrolladores que no conocen el patrón: el flujo no es evidente leyendo solo
  la clase de servicio.
- Si en el futuro hubiera solo un tipo de pedido, la Factory y la interfaz
  agregarían indirección sin beneficio (señal de over-design según sección 4.1).
