# ADR-003: Spring Events (Observer) para notificaciones desacopladas

## Estado

Aceptado — 2026-05-16

## Contexto

Cuando un pedido es procesado exitosamente, el sistema debe notificar a múltiples
receptores. En la versión legacy, `ServicioPedidosLegacy` tenía acoplamiento directo
a `JavaMailSender` inyectado con `@Autowired`. Esto generaba tres problemas:

1. **Acoplamiento efferente directo:** el servicio central dependía de la implementación
   concreta del canal de email, elevando su CE y su fragilidad ante cambios.
2. **Dificultad de extensión:** agregar un canal de log o SMS requería modificar el
   servicio de pedidos (viola OCP).
3. **Imposibilidad de prueba sin mock de JavaMail:** la prueba del flujo de negocio
   arrastraba la dependencia del servidor SMTP.

Se evaluaron tres alternativas:

- **Alternativa A (acoplamiento directo):** inyectar todos los canales en el servicio
  — fácil de leer, pero el CE crece con cada canal nuevo.
- **Alternativa B (lista de ServicioNotificacion):** inyectar `List<ServicioNotificacion>`
  y llamarlos en bucle — reduce el acoplamiento pero el servicio aún conoce los canales.
- **Alternativa C (Spring Events):** publicar un evento de dominio y dejar que cada
  canal suscriba de forma independiente con `@EventListener`.

## Decisión

Se elige la **Alternativa C: Spring ApplicationEventPublisher (Observer)**.

- Se define `PedidoProcesadoEvent` como un `record` inmutable en la capa de aplicación.
- `FachadaPedidos` publica el evento mediante `ApplicationEventPublisher.publishEvent()`.
- `NotificacionEmail` y `NotificacionLog` implementan `ServicioNotificacion` y suscriben
  el evento con `@EventListener`, de forma independiente entre sí.
- El publicador no conoce a los suscriptores. Agregar un canal SMS requiere únicamente
  crear una nueva clase `@Component` con `@EventListener`, sin tocar la Facade.

## Consecuencias

**Positivas:**
- Agregar un nuevo canal de notificación no requiere modificar ninguna clase existente
  (cumple OCP de forma estricta).
- Los listeners son opcionales: si ninguno está registrado, el evento se descarta
  sin error ni excepción.
- `FachadaPedidos` tiene Cognitive Complexity = 0: el flujo principal no tiene ramas
  condicionales relacionadas con notificaciones.
- El dominio y la capa de aplicación son testeables sin ningún canal concreto activo.

**Negativas:**
- El orden de ejecución de los listeners no está garantizado. Si en el futuro dos
  listeners tuvieran una dependencia de orden (primero auditoría, luego email),
  habría que usar `@Order` o migrar a un mecanismo con garantía de secuencia.
- El debugging es más complejo: al ver `publishEvent()` en la Facade, no es evidente
  qué listeners responderán sin buscar todas las clases con `@EventListener`.
- El acoplamiento entre `PedidoProcesadoEvent` (en `aplicacion`) y `ServicioNotificacion`
  (en `dominio.puertos`) crea una dependencia leve desde el puerto hacia la capa de
  aplicación, que debe monitorearse para no invertir la dirección de dependencias.
