# ADR-001: Adopción de Arquitectura Hexagonal para aislar el dominio

## Estado

Aceptado — 2026-05-16

## Contexto

El sistema de gestión de pedidos debe soportar múltiples tipos de procesamiento
(Estándar, Express, Internacional) y múltiples canales de notificación (email, log,
y potencialmente SMS o push en el futuro). La implementación inicial
(`ServicioPedidosLegacy`) acoplaba directamente el dominio a `Spring Data JPA`
y `JavaMailSender`, lo que generaba los siguientes problemas:

- Las pruebas unitarias del servicio requerían levantar el contenedor de Spring
  y una base de datos embebida, incrementando el tiempo de feedback.
- Cambiar de JPA a MongoDB implicaría modificar clases de dominio.
- Agregar un canal de notificación requería modificar el servicio central.
- El acoplamiento efferente (CE) del servicio era 6, con Instability = 1.00,
  indicando un núcleo frágil ante cualquier cambio de infraestructura.

Las métricas de acoplamiento de Robert C. Martin (sección 3.1 de la guía)
confirman que el paquete de dominio debe ser el más estable del sistema
(Instability → 0), lo cual es incompatible con dependencias a JPA o JavaMail.

## Decisión

Se adopta la **Arquitectura Hexagonal** (Ports & Adapters). Las reglas son:

1. El paquete `dominio` define **puertos** (interfaces) y **entidades** de negocio.
2. El paquete `infraestructura` contiene implementaciones técnicas (JPA, notificaciones).
3. El paquete `adaptadores` contiene las estrategias de procesamiento, la Facade y el controlador REST.
4. El dominio **no importa** ninguna clase de `org.springframework`, `jakarta.persistence`
   ni de ningún otro paquete de infraestructura.
5. La regla arquitectónica `dominioAisladoDeInfraestructura` en `ReglasArquitecturaTest`
   verifica esta restricción automáticamente en cada build.

## Consecuencias

**Positivas:**
- El dominio es testeable con Mockito puro, sin contenedor de Spring ni base de datos.
- Cambiar de JPA a MongoDB requiere únicamente crear un nuevo adaptador en `infraestructura`,
  sin tocar ninguna clase de dominio.
- El Instability del paquete `dominio` pasa de 1.00 a 0.00 tras la refactorización.
- Las decisiones de diseño son verificables automáticamente con ArchUnit en CI/CD.

**Negativas:**
- La estructura tiene más archivos que un MVC clásico: cada puerto requiere al menos
  una interfaz y una implementación separada.
- La curva de aprendizaje es mayor para desarrolladores acostumbrados a Spring MVC
  sin separación de puertos.
- Los mappers entre entidades de dominio y entidades JPA añaden código que en un
  sistema simple podría considerarse sobreingeniería (YAGNI).
